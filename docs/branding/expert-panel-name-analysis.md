# Expert Panel Name Analysis

## Executive Summary

Final platform name: `Diognosis`.

This document records the final naming decision after the expert-panel naming review. The project was previously developed under working names including `MedCheck` and a short-lived platform rename. The final master brand is now `Diognosis`, with `MedCheck Engine` preserved as the first medication-safety module.

This document does not rename files by itself; it records the brand architecture now used across the repository.

Important disclaimer: this is not legal advice, not a trademark clearance search, and not a domain availability report. Final trademark, domain, GitHub, npm, LinkedIn, and search checks must still be completed outside the repository before treating the name as fully cleared.

## Current Repository Naming State

Current intended naming model:

- Platform/project: `Diognosis`
- First module/product: `MedCheck Engine`
- Public formulation: `Diognosis — MedCheck Engine`
- Alternative formulation: `MedCheck Engine by Diognosis`
- Package slug: `diognosis`
- Expected GitHub Pages path after repository rename: `https://diogonmpacheco.github.io/Diognosis/`

Repo content should use `Diognosis` when referring to the platform, project, repository, browser title, SEO title, live app, public docs, GitHub links, package name, and LinkedIn-facing identity.

Repo content should keep `MedCheck Engine` when referring to the specific medication-safety module, drug interaction checking, pharmacogenomics logic, medication review workflow, engine internals, or compatibility identifiers such as `MEDCHECK_*`.

## Evaluation Method

Scores are `1-5`, where `5` is strongest/best.

For `Low generic/confusion risk`, a `5` means lower risk and stronger distinctiveness; a `1` means the name is generic, crowded, confusing, or already visibly used in adjacent healthcare/pharma contexts.

Criteria:

- `Distinctiveness`: how ownable and memorable the name feels.
- `SEO potential`: likelihood of ranking for exact-name and project-specific searches.
- `Healthcare credibility`: whether it sounds serious in medication safety/pharmacogenomics.
- `Recruiter/CV value`: whether it supports a serious LinkedIn/project narrative.
- `Expansion potential`: whether it can support future modules beyond medication checking.
- `Low generic/confusion risk`: preliminary risk posture before legal/domain checks.
- `Fit with current project`: how well it describes the current MedCheck Engine.
- `Fit with future platform`: how well it supports evidence graphs, PGx, metabolites, pathway maps, toxicity registries, and patient-led research tools.

## Expert Panel Review

### 1. Brand Strategist

Recommended architecture: branded house.

`Diognosis` should be the master brand. `MedCheck Engine` should remain the first module name because it is concrete, understandable, and already attached to the current medication-safety tool. This avoids forcing the broader platform to sound like only a drug checker.

The spelling `Diognosis` is distinctive and personal. It reads as a deliberate blend rather than a generic medical word, while still evoking diagnosis, discovery, and Diogo-led patient research.

### 2. SEO Strategist

`Diognosis` has strong exact-name SEO potential because it is uncommon and visually distinct from the generic word `diagnosis`. The main risk is misspelling or autocorrection by users and search engines. This can be managed by consistent metadata, repeated exact-name usage, and page copy that explains the name without pretending it is the ordinary medical term.

Recommended SEO pairing:

- `Diognosis medication safety`
- `Diognosis pharmacogenomics`
- `Diognosis drug interactions`
- `Diognosis MedCheck Engine`
- `Diognosis active metabolites`
- `Diognosis CYP2D6`

### 3. Trademark / Naming Risk Reviewer

`Diognosis` is more distinctive than generic candidates such as `DrugAtlas`, `EvidenceGraph`, or `MedTrace`. It may still be phonetically close to `diagnosis`, so external screening must check for healthcare, diagnostics, AI health, clinical decision-support, and medical software names that sound similar.

Risk posture: promising but not cleared.

Required external checks:

- Exact `Diognosis`
- Similar spellings: `Diagnosys`, `Dignosis`, `Dionosis`, `DiagnoSys`, `DiognoSys`
- Ordinary-word confusion with `diagnosis`
- Healthcare AI, diagnostics, clinical software, pharmacogenomics, medication safety, and patient research categories

### 4. Healthcare Product Reviewer

`Diognosis` is credible if the product copy stays cautious. Because the name is close to `diagnosis`, public disclaimers must remain strong:

- Not medical advice
- Not a clinical decision support system
- Pre-v1 research prototype
- Source-linked evidence does not equal clinical validation
- Pending future professional clinical review

The platform must not imply it diagnoses patients. The best framing is medication-safety and pharmacogenomics exploration, not diagnosis or prescribing.

### 5. Recruiter / Hiring Manager

`Founder & Product Lead — Diognosis` is stronger and more personal than a generic healthcare name. It gives the project a founder-led identity and can help explain a CV gap as active product, research, and systems-design work.

Recommended LinkedIn role:

`Founder & Product Lead — Diognosis`

Recommended LinkedIn description:

`Building Diognosis, a source-linked medication safety and pharmacogenomics platform. Its first module, MedCheck Engine, explores drug interactions, pharmacogenomics, metabolites, PK exposure shifts, transporter pathways, and source-linked evidence.`

### 6. Open-Source Maintainer

Use lowercase `diognosis` for repository and package naming. It is easy to type once learned, but README copy should make the spelling unmistakable.

Recommended forms:

- Human brand: `Diognosis`
- Repository slug: `Diognosis`
- Package name: `diognosis`
- Browser short name: `Diognosis`
- Module name: `MedCheck Engine`

Avoid introducing extra spellings such as `DioGnosis`, `Dio-gnosis`, or `DiogNosis`. Consistency matters.

### 7. Technical Architect

`Diognosis` can support future modules beyond MedCheck Engine:

- Evidence graph explorer
- PGx report bridge
- Metabolite intelligence
- Pathway maps
- Toxicity registries
- Medication-class discovery views
- Patient-led research workflows
- Professional review queues

The name is broad enough to hold patient-led research, medication safety, evidence modeling, and future modules without forcing every product to include `Med`, `Rx`, or `Pharm`.

## Rejected Or High-Risk Names

These names should not be used without strong external clearance. Several should probably be rejected outright:

- `RxTrace`: high-risk. RxTrace has an existing healthcare/pharma traceability site, and the site states "RxTrace is a registered trademark. All rights reserved." See [rxtrace.com](https://www.rxtrace.com/).
- `OpenPGx`: high-risk. OpenPGx already exists as an open pharmacogenomics project/site. See [openpgx.ai](https://openpgx.ai/) and [GitHub open-pgx/openpgx](https://github.com/open-pgx/openpgx).
- `MedTrace`: high-risk. MedTrace exists as a pharma/diagnostics company. See [medtracepharma.com](https://medtracepharma.com/).
- `ClinTrace`: high-risk. ClinTRACE appears as a CTMS product, and Clintrace also appears in adverse-event/product-safety contexts. See [clintrace.com](https://clintrace.com/) and [Pharmaceutical Online Clintrace](https://www.pharmaceuticalonline.com/doc/clintrace-0001).
- `RxLens`: high-risk. RxLens has existing pharmacy/AI usage in a 2025 Amazon Science publication. See [Amazon Science RxLens](https://www.amazon.science/publications/rxlens-multi-agent-llm-powered-scan-and-order-for-pharmacy).
- `PharmGraph` / `PharmaGraph`: high-risk. These are too close to Pharmagraph, an existing pharma/life-sciences monitoring company. See [pharmagraph.co.uk](https://pharmagraph.co.uk/).
- `SignalRx`: high-risk. SignalRx appears as a pharmaceutical company name in oncology drug-development context. See [UT Health San Antonio news](https://news.uthscsa.edu/mays-cancer-center-researcher-and-signalrx-earn-nci-business-grants-to-develop-drugs-for-first-in-human-clinical-trials/).
- `InteractRx`: high-risk. interactRx appears in existing healthcare/life-sciences tools. See [Deloitte interactRx](https://www.deloitte.com/us/en/industries/life-sciences-health-care/about/interactrx-fair-market-value-physician-compensation-tool.html).
- `DoseTrace`: high-risk. DoseTrace has existing medication tracking and DSCSA/traceability usage. See [dosetrace.io](https://dosetrace.io/) and [dosetrace.org](https://dosetrace.org/).
- `MetaboMap`: caution. MetaboMap appears in metabolic testing/research contexts. See [Boston Heart Serum MetaboMap PDF](https://learnbhd.com/s/serum-metabomap.pdf).
- `EvidenceGraph`: weak as a brand. It is an excellent technical descriptor but likely too generic to own.
- `DrugAtlas`: weak as a brand. Clear but generic, and it narrows the platform to drug lookup rather than evidence/pathway modeling.
- `SourceGraph`: weak fit. It sounds like a general software/graph technology name, not a medication-safety/pharmacogenomics platform.
- `SafeRxTrace`: caution. It implies safety and may overclaim for a pre-v1, not-professionally-reviewed research prototype.

## Recommended Naming Architecture

Recommended architecture: branded house.

- Platform/project: `Diognosis`
- First module/product: `MedCheck Engine`
- Public formulation: `Diognosis — MedCheck Engine`
- Alternative formulation: `MedCheck Engine by Diognosis`
- Legacy note: `MedCheck` remains historical/internal only unless referring to `MedCheck Engine`

Reasoning:

- `Diognosis` is broad enough for future modules and distinctive enough to build a founder-led identity.
- `MedCheck Engine` preserves continuity with the original project and describes the current module clearly.
- The architecture avoids presenting `MedCheck` as the company/project name.
- The architecture reduces future rename pressure if the platform expands into evidence graphs, PGx report tooling, metabolite intelligence, pathway maps, toxicity registries, or patient-led research tools.

Suggested wording:

`Diognosis is a source-linked medication safety and pharmacogenomics platform in active development. Its first module, MedCheck Engine, explores drug interactions, pharmacogenomics, active and toxic metabolites, PK exposure shifts, transporter pathways, medication class effects, and source-linked evidence through a privacy-preserving static web application.`

Status line:

`Status: pre-v1, source-linked, under active validation, and not yet professionally reviewed.`

## Required External Checks Before Rename

Before treating the name as cleared, run a real external clearance pass:

- Trademark search in relevant jurisdictions, especially U.S., EU, U.K., and any intended launch geography.
- Domain checks for `.com`, `.org`, `.dev`, `.ai`, and practical alternatives.
- GitHub organization/repository availability check.
- npm package availability check.
- LinkedIn company/page and experience-name search.
- Google exact-match search for `"Diognosis"`, `"Diognosis medication safety"`, `"Diognosis pharmacogenomics"`, and `"Diognosis drug interactions"`.
- Search for confusingly similar forms such as `Diagnosis`, `Diagnosys`, `Dignosis`, `Dionosis`, `DiagnoSys`, `DiognoSys`, and `Diognosis Health`.
- Healthcare/pharma/life-sciences database search for companies, products, CRO tools, clinical trial systems, pharmacy software, diagnostics, clinical decision support, and pharmacovigilance products.

Do not treat this document as final legal clearance. It records the selected name and the checks still required.

## Final Recommendation

Use `Diognosis` as the final platform/project name and keep `MedCheck Engine` as the first module name.

The recommended public architecture is:

`Diognosis — MedCheck Engine`

Do not use `MedCheck` alone as the company/project/platform name. Keep it as `MedCheck Engine`.

Final external clearance remains required before launch claims, domain registration, LinkedIn company-page creation, or broader public promotion.
